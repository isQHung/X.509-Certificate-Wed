# PoC Note

## 0. How to Implement & Setup
Each PoC needs to have installation instructions and purpose in the README.md.


## 1. Clearly define the objective 🎯

Before building anything, specify **what you are trying to prove**.

Examples:

* Can this AI model detect fraud accurately?
* Can this new database handle 10k requests per second?
* Can this API integrate with our existing system?

Rule: **One PoC = one main question to answer.**

---

## 2. Define success criteria 📊

You must know **what success looks like**.

Examples:

* Response time < 200 ms
* Accuracy > 90%
* Integration works without major architecture changes

Without measurable criteria, a PoC becomes meaningless.

---

## 3. Keep the scope minimal ✂️

A PoC is **not a full product**.

Rules:

* Build **only the essential features**
* Avoid UI or production-level security unless required
* Use mock data if possible

Goal: **prove feasibility, not deliver a finished system.**

---

## 4. Use quick and flexible technology 🧰

Choose tools that allow **rapid development**, such as:

* scripting languages
* cloud services
* simple frameworks

The PoC code **does not need to be production quality**.

---

## 5. Document assumptions and risks 📄

Write down:

* assumptions you are testing
* technical limitations
* dependencies

This helps stakeholders understand **what the PoC proves and what it doesn't.**

---

## 6. Test and collect evidence 🧪

Run experiments and gather data such as:

* performance metrics
* logs
* screenshots
* test results

A PoC should **show clear evidence**, not just opinions.

---

## 7. Evaluate and decide 🚦

After testing, conclude one of three outcomes:

* ✅ **Feasible** – move to prototype or development
* ⚠️ **Needs changes** – adjust architecture or tools
* ❌ **Not feasible** – stop or rethink the approach

---

💡 **Good PoC characteristics:**

* Fast (days or weeks)
* Low cost
* Limited scope
* Data-driven results

---
